# Narrative Studio 叙事工作台：项目构想整理

## 一、项目定位

Narrative Studio 是一款以中文自然语言为核心的互动叙事创作网站。

它主要用于创作：

- 文字冒险游戏；

- 视觉小说；

- 多视角互动故事；

- 多角色群像叙事；

- 可以切换角色操纵的剧情游戏；

- 以文本、选择和状态变化为核心的实验性游戏。

作者可以直接在网站中创作和试玩作品，也可以将作品转换成 Godot 可以读取的数据，在 Godot 中继续开发。

它不是传统代码编辑器，也不只是剧情流程图工具。

它更接近：

> 一个以自然语言组织世界、人物、视角、选择和叙事效果的互动故事引擎。
> 
>

# 二、目前已经形成的核心设计

## 作品的基本结构

目前可以暂定为：

```Plain Text
作品
└── 段落
意象群
```

### 作品

一个完整的互动故事或游戏项目。

例如：

```Plain Text
《雾港》
```

### 段落

作品中可以相对独立发布、试玩和分享的一部分。

这里的“段落”不是普通文章中的自然段，而是一个具有一定完整性的互动故事单位。

一个段落可以包含：

- 多个角色；

- 多个场景；

- 多种视角；

- 多个意象；

- 选择和效果；

- 初始世界状态；

- 结束条件；

- 使用的系统。

“段落”这个名称目前可以暂时使用，但以后也可以考虑“篇章”“剧段”或“剧局”。

### 意象

意象是一段长度不限的自然语言文本。

它可以是一句话，也可以是一段很长的描述、对话、记忆或事件。

例如：

> 雨水沿着候车室的玻璃向下滑。林默想起十年前也是这样的雨，只是那时坐在对面的人还愿意看着他。
> 
> 

整个语段就是一个意象。

### 意象群

多个在意义、情绪、人物、时间或事件上相互接近的意象，可以构成一个意象群。

意象群不是文件夹，也不一定拥有严格的父子关系，而是一组在空间中相互靠近的意象点。

---

## 意象空间

每个意象在空间中表现为一个点。

它拥有三个坐标：

```Plain Text
意象 =（X，Y，Z）
```

其中：

- X 坐标由意象的文本长度自动决定；

- Y 坐标由作者调整；

- Z 坐标由作者调整。

作者主要在 YOZ 平面上组织意象之间的关系。

从正面看，它是一个平面上的意象点分布；将视角倾斜以后，可以看到由文本长度形成的 X 轴纵深。

为了避免长文本距离过远，X 坐标不能直接等于字数，可以对字数进行压缩处理。

例如：

```Plain Text
10 字：X = 1
100 字：X = 2
1000 字：X = 3
10000 字：X = 4
```

用户不需要看到具体计算过程，只需要感受到：

> 越长的意象，在空间中的纵深越大。
> 
> 

### 空间操作

网站可以参考《邪恶冥刻》的镜头移动方式

- 上下左右：在附近的意象、角色或区域之间移动；

---

## 不同视角

作品可以不存在固定主角。

同一个段落中可以有：

- 角色视角；

- 旁白视角；

- 导演视角；

- 多个可操纵角色；

- 没有归属人物的观察视角。

玩家可以在不同角色之间切换。

同一件事情，在不同视角下可以展示完全不同的文本和选项。

例如，真实世界中发生的是：

> 林默将信收进了口袋。
> 
> 

林默视角可能看到：

> 不能让她看到信的最后一行。
> 
> 

苏禾视角可能看到：

> 他收信时太匆忙了，像是在掩饰什么。
> 
> 

导演视角则看到：

> 林默将信收进右侧口袋。苏禾看见了他的动作，但没有看清信件内容。
> 
> 

因此，视角切换不是简单更换头像，而是改变：

- 能看到什么；

- 如何描述看到的事情；

- 可以选择什么；

- 当前知道什么；

- 当前能够影响什么。

---

## 角色的三个基本维度

角色和叙事目前围绕三个维度展开：

### 导演世界

导演世界代表客观存在和游戏运行层面的内容。

它可以包括：

- 人物；

- 物品；

- 地点；

- 时间；

- 天气；

- 已经发生的事件；

- 人物和物品的位置；

- 当前存在的按钮；

- 可以进入的场景；

- 可以执行的交互；

- 公开的世界规则。

角色不一定知道导演世界中的全部内容。

### 自身能力

能力表示角色可以如何感知或改变世界。

能力不一定只是数字，还可以是：

- 数值；

- 技能；

- 身份；

- 权限；

- 身体状态；

- 知识；

- 特殊性质；

- 可以执行的行为。

每项能力可能具有作用对象。

例如：

```Plain Text
能力：火焰魔法

可能作用于：
- 人物
- 可燃物
- 黑暗空间

使用条件：
- 拥有魔力
- 目标在视野内
```

能力需要区分：

```Plain Text
角色是否拥有能力
当前世界是否允许使用
是否存在可以作用的对象
使用后是否能够产生效果
```

例如魔法师穿越到一个没有魔力的世界，作者可以决定：

- 魔法完全失效；

- 魔法可以使用，但无法恢复魔力；

- 魔法被转换成另一种力量；

- 魔法可以发动，但找不到合适的作用对象；

- 魔法被当地人理解为神迹或妖术。

这些都不是系统自动决定的，而是作者可以选择和描述的内容。

### 自身欲望

欲望表示角色为什么观察、选择和行动。

欲望通常不适合简单地变成数值。

它更适合使用自然语言表达：

> 她希望林默主动承认错误，但又不愿直接询问，因为那等于承认自己仍然在乎。
> 
> 



或者不表达，作者自己知道就可以

---

## 选择和效果

一个选项不能只代表“跳转到下一段文字”。

它可以产生多种效果：

- 改变世界状态；

- 改变人物位置；

- 改变时间；

- 增加或削弱能力；

- 获得物品；

- 改变某个角色的欲望描述；

- 向另一个角色发送事件；

- 解锁另一个角色的选项；

- 切换视角；

- 改变按钮、背景、音乐或叙事方式。

为了让不同系统能够组合，可以统一使用下面的结构：

```Plain Text
事件
条件
动作
状态
```

也就是：

> 当某件事情发生，如果满足某些条件，就执行一组动作，并改变某些状态。
> 
> 

例如：

```Plain Text
当：苏禾选择“打开信件”

如果：
- 信件仍然存在
- 苏禾拥有信件
- 苏禾能够阅读文字

执行：
- 苏禾获得信件内容
- 林默的秘密被苏禾知道
- 苏禾获得“质问林默”的选项
```

不同系统内部可以完全不同，但需要按照相同方式声明：

- 它保存什么状态；

- 它能接收什么动作；

- 它会发出什么事件；

- 它需要哪些条件；

- 它支持网站、Godot，还是同时支持两者。

---

## 可共享的系统

未来可以允许用户在社区中发布系统，例如：

- 旁白系统；

- 按钮系统；

- 对话框系统；

- 好感度系统；

- 背包系统；

- 时间系统；

- 手机聊天系统；

- 掷骰系统；

- 调查系统；

- 多角色控制系统。

普通用户不需要接触代码。

所有人编写系统的方式都是一样的，通过网站的选项来设置

---

## 中文自然语言界面

网站以中文自然语言为主。

它不应该要求普通创作者使用：

- 编程语言；

- 大量图形节点；

- 复杂表格；

- 特殊脚本语法；

- 一堆关键词占位符。

整体风格应当是：

- 大量留白；

- 文字优先；

- 界面安静；

- 工具栏尽量隐藏；

- 操作产生空间反馈；

- 很少出现强制弹窗；

- 不让功能压过创作本身。

后台可以保留结构化数据，但作者看到的应当仍然是自然语言。

---

## 提问系统

网站可以在适当的时候提出问题。

例如：

> 伊莱的火焰魔法在这个世界仍然可以使用吗？
> 
> 

这些问题的作用是提醒作者可能尚未确定的关系，而不是要求作者回答。

用户可以：

- 回应；

- 略过；

一个问题展示一次后就不会重复出现。

第一版不需要连接 AI，可以采用规则触发。

例如：

```Plain Text
当角色进入一个新世界
并且角色拥有某项能力
但能力与世界规则的关系尚未确定
则询问一次：
“这个角色的这项能力在这里仍然可以使用吗？”
```

用户回答后，什么也不会发生

必须保持克制：

- 一次只问一个问题；

- 通常不超过两行；

- 不评价写得好不好；

- 不主动续写长篇内容；

- 不连续追问；

- 用户略过后不再骚扰；

- 允许故事保留矛盾和空白。

---

## 网站试玩与 Godot 导出

编辑器、网站试玩和 Godot 不能分别使用三套剧情数据。

它们应该共享一种统一的作品数据：

```Plain Text
Narrative Studio 编辑器
              ↓
         统一作品数据
          ↙         ↘
    网站运行器     Godot 运行器
```

网站运行器读取数据并在浏览器中试玩。

Godot 运行器读取同一份数据，并将文本、角色、按钮、选项和效果表现为 Godot 场景。

第一版不一定需要自动生成完整 Godot 项目。

更现实的方式是：

- 导出统一 JSON；

- 提供一个 Godot 运行插件或示例工程；

- Godot 读取 JSON；

- 根据 JSON 展示对话和选项。

# 三、目前最有辨识度的部分

Narrative Studio 目前最独特的地方不是单独某个功能，而是下面几个设计结合在一起：

1. 中文自然语言优先；

2. 不要求创作者编程；

3. 可以切换角色视角；

4. 世界、能力和欲望相互影响；

5. 同一事件在不同角色眼中拥有不同文字；

6. 作者可以在意象空间中组织语段；

7. 网站会在留白处提出一次性问题；

8. 作品可以在网页试玩，也可以接入 Godot；

9. 系统和外观未来可以在社区共享。

可以将产品理念概括为：

> 不是只决定接下来发生什么，而是决定由谁看见、如何理解，以及他能够怎样改变正在发生的一切。
> 
>
