问题2：有没有权限有什么区别？怎么获得权限？
#

获取权限的方法：

场景操作方法说明与警告
临时获得一次最高权限以管理员身份运行当前程序1. 找到程序（如文件资源管理器、命令行）
2. 右键点击其图标 → 选择"以管理员身份运行"
3. 在弹出的UAC窗口中点击"是"
此后，这个窗口内的操作暂时拥有最高权限
永久为自己添加对某个文件夹的权限手动修改文件夹的安全属性1. 右键点击目标文件夹 → “属性”
2. 切换到"安全"选项卡
3. 点击"编辑" → “添加” → 输入你的用户名 → 勾选"完全控制" → 确定
⚠️ 警告：对系统目录这样做会降低安全性，强烈不推荐
一劳永逸避免权限问题将软件安装/放置在你已有完全控制权的个人文件夹中例如，在C:或D:盘根目录下新建一个Tools或Dev文件夹。你在此类自己创建的文件夹中天生拥有"完全控制权"，永远不需要额外获取权限

问题3：所以有权限就等于有路径吗？
#

答案：不等于

条件结果原因分析
有权限，无路径在任何地方输入hugo version，会显示"命令未找到"系统不知道hugo这个命令在哪里。你有权操作文件，但系统找不到它
有路径，无权限可以运行hugo version，但创建文件等操作时可能失败系统知道命令在哪，但当你试图在受保护的目录里创建、修改文件时，会被权限墙挡住

✅ 结论：你需要同时满足两个条件

  1. 位置上有权限：把hugo.exe放在一个你天生拥有"完全控制权"的文件夹里（如C:Tools、D:Hugo）
  2. 系统中有路径：把上述文件夹的路径添加到系统的PATH环境变量

对比项npx blowfish-tools（临时调用工具）winget install Git.Git（安装系统软件）
核心动作临时调用并执行永久安装到系统

问题2：为什么安装了梯子之后，Git还需要经过配置才能使用呢？
#

配置本质：

git config --global http.proxy http://127.0.0.1:6666
git config --global https.proxy http://127.0.0.1:6666

或使用SSH（git@github.com:…）绕过HTTP代理问题。

配置的作用：

这两条命令告诉Git：

  • 通过HTTP协议访问远程仓库时，使用指定的代理服务器
  • 通过HTTPS协议访问远程仓库时，同样使用指定的代理服务器

具体参数含义：

git config --global http.proxy http://127.0.0.1:6666
git config --global https.proxy http://127.0.0.1:6666
  • git config：用来设置、查看、修改Git配置的核心命令。Git的所有行为都通过这些配置来控制
  • --global：全局配置，对当前用户的所有仓库生效
  • http://127.0.0.1:6666：代理服务器地址和端口
  • 127.0.0.1：本地回环地址（localhost）
  • :6666：代理软件监听的端口（如1080、7890等），这是从你的设置→代理里面获得的，并不是机场给的那个端口

简单来说： 需要GitHub到电脑，电脑的梯子（也就是那个软件）来处理，转发给机场的代理服务器。机场给的端口就是这个远程的代理服务器端口。GitHub并不知道怎么上远程服务器（没有加密和协议转换），当然是连接不到的。

问题3：SSH是什么？
#

我已经知道我需要从GitHub的settings去生成一个，SSH同时似乎代表着一个在C盘的文件和一串代码，这个为什么可以绕过HTTPS呢，有什么不同？什么好处吗？

SSH基本概念：

SSH（Secure Shell）是一个加密的网络协议，用于安全地远程访问和管理计算机。在Git中，SSH提供了另一种访问GitHub的方式。

SSH的"三个部分"：

  1. SSH密钥对（C盘的文件）
# 通常位置
~/.ssh/id_rsa       # 私钥（私有的，绝不给别人）
~/.ssh/id_rsa.pub   # 公钥（公开的，给GitHub）
  • 私钥：在你电脑上，证明"你是你"
  • 公钥：给GitHub，用来验证你的私钥
  1. GitHub配置（settings中的配置）
  • 把公钥内容复制到GitHub → Settings → SSH and GPG keys
  • GitHub会说：“好的，这个公钥对应xx用户”
  1. SSH地址（一串代码）

格式：[git@github.com](mailto:git@github.com):用户名/仓库.git

SSH vs HTTPS的本质区别：

方式协议认证方式端口特点
HTTPSHTTP协议 + SSL/TLS加密每次操作都需要用户名+密码（或token）443端口容易被网络代理/防火墙拦截
SSHSSH协议使用密钥对认证（一次配置，永久使用）22端口更难被拦截

SSH设置全流程：

第1步：生成SSH密钥

# 检查是否已有
ls -al ~/.ssh

# 生成新密钥（一路回车）
ssh-keygen -t rsa -b 4096 -C "your_email@example.com"

# 或更简单的
ssh-keygen -t ed25519 -C "your_email@example.com"

第2步：复制公钥到GitHub

# Windows
cat ~/.ssh/id_ed25519.pub
# 或直接打开文件 C:\Users\你的用户名\.ssh\id_ed25519.pub

# 复制全部内容，包括开头的ssh-ed25519和结尾的邮箱

第3步：添加到GitHub

  1. 登录GitHub → Settings
  2. SSH and GPG keys → New SSH key
  3. 粘贴公钥，取个名字（如"My Laptop"）

第4步：测试连接

ssh -T git@github.com
# 看到这个表示成功：
# Hi 用户名! You've successfully authenticated...

为什么SSH可以绕过HTTP代理配置？

HTTP/HTTPS方式：

# 需要明确告诉Git代理地址
git config --global http.proxy http://127.0.0.1:7890

翻译：“Git，请把所有HTTP流量发给127.0.0.1:7890这个代理”

SSH方式：

# 什么都不用配置！直接使用SSH地址
git clone git@github.com:username/repo.git

Git自动知道：“哦，这是SSH地址，我该用系统的SSH客户端去连接”

工作流程对比：

HTTP/HTTPS流程（需要配置代理）：

[你的Git命令]
    ↓
[Git程序]
    ↓ 检查配置：发现http.proxy=http://127.0.0.1:7890
[发送请求到127.0.0.1:7890]
    ↓
[代理软件] ← 正在监听7890端口！
    ↓
[代理软件处理请求]
    ↓
[转发到机场服务器]
    ↓
[机场服务器访问GitHub]

SSH流程（什么都不用配）：

你的Git命令
    ↓
[Git] → "我要访问git@github.com..."
    ↓
检查地址格式 → 是SSH格式！
    ↓
直接调用系统SSH客户端
    ↓
[系统SSH] → 连接github.com:22
    ↓
使用~/.ssh/id_rsa私钥认证
    ↓
成功连接，传输数据

SSH也需要配代理（如果SSH被墙）：

需要在~/.ssh/config中添加：

Host github.com
    ProxyCommand nc -X 5 -x 127.0.0.1:7890 %h %p

操作方法：在SSH的统一文件夹里面新建一个config文本文件，里面写：

Host github.com
    ProxyCommand nc -X 5 -x 127.0.0.1:7890 %h %p

总结要点：

  1. HTTPS方式：需要配置Git代理，走443端口，容易被拦截
  2. SSH方式：使用密钥认证，走22端口，更难被拦截
  3. SSH配置：一次配置永久使用，Git里不需要额外配置
  4. 代理原理：HTTPS代理在Git中配置，SSH代理在SSH配置文件中配置
  5. 核心区别：HTTPS需要每次验证，SSH一次配置永久使用

💡 DS的解答：

错误原因分析：

错误提示让你手动运行git submodule add...，这说明问题出在Git操作上，而不是网络或Hugo。

结合你的记忆，根本原因很可能是这两点之一或同时发生：

  • 原因A：缺少hugo.exe（你的记忆）：blowfish-tools在创建Hugo站点后，需要立即运行一次hugo命令来验证。如果系统PATH里没有hugo，或者项目目录里没有hugo.exe，这个验证失败就会导致整个流程中断，还没执行到添加主题那步就报错了。你复制hugo.exe进来，实际上是为这个"验证"步骤提供了本地可执行文件。
  • 原因B：Git未正确初始化（你的猜测）：git submodule add命令必须在Git仓库目录中执行。虽然工具显示√ Git initialized，但如果初始化过程被中断或未在正确的子目录（newSite）中完成，也会导致后续步骤失败。

git init到底在干嘛？它应该在哪一级？

  • 作用：git init会在你执行该命令的当前目录下，创建一个名为.git的隐藏文件夹。这个文件夹就是Git仓库的"数据库"，记录所有版本历史。没有它，任何git命令（包括submodule add）都无法执行。
  • 应该在哪一级：必须在你的Hugo博客项目根目录下执行。
    • 错误的位置：在D:或包含很多其他文件的目录下运行git init
    • 正确的位置：进入你的项目文件夹（例如D:newSite），然后在这里运行git init。这样，.git文件夹就会创建在D:newSite.git\
  • 如何检查：打开你的newSite文件夹，在"查看"选项中勾选"隐藏的项目"。如果里面有一个.git文件夹，说明Git已正确初始化。如果没有，就是原因B。

关于"复制hugo.exe"：

教程里嘱咐复制hugo.exe到项目文件夹，是一个古老的、临时的变通方法，并不是标准做法。标准且一劳永逸的做法是：将hugo.exe所在目录（如C:Hugo）添加到系统PATH环境变量。这样，你在电脑的任何地方（包括newSite文件夹）输入hugo，系统都能找到它，完全不需要复制。

你复制后解决了问题，恰好证明了当时系统PATH没配好，工具无法调用全局的hugo命令。

✅ 标准流程重做：

  1. 确保Hugo和Git在全局可用（在任意文件夹打开命令行都能运行）：
hugo version
git --version

两者都必须成功显示版本号。如果hugo报错，回去将hugo.exe的目录添加到系统PATH。

  1. 用blowfish-tools new一键创建（先安装或者拉取tools）：
# 1. 先进入你希望创建项目的父目录（比如D盘根目录）
cd /d D:\

# 2. 运行一键创建命令，myblog是你的项目名

💡 DS的解答：

当blowfish-tools选择Configure menus时，它的工作逻辑是：

  1. 目标明确：它要在你的博客项目根目录（即newsite）下创建或修改配置文件

你的newsite项目/
└── config/              <-- 工具要操作的是这里
    └── _default/
        └── menus.en.toml  <-- 工具要创建或修改的文件在这里

结论与启示：

  1. 你没错，工具也没错：你之前手动看到的是主题的示例，而工具要创建的是你站点的实际配置。这是两个不同的文件，放在两个不同的地方

💡 DS的解答：

为什么用content/pages会失败，而content/docs会成功？

这完全由Blowfish主题的配置决定的，和Hugo本身无关。

  • docs是主题预设的"内容类型"：Blowfish主题在它的模板中，已经定义好了一个名为docs的内容类型（Content Type）。当你创建content/docs/[xxx.md](http://xxx.md)时，主题知道该用哪套布局（比如侧边栏导航、特定的页面样式）来渲染它。你从示例里抄来的配置，正好激活了这个类型
  • pages不是预设类型：B站教程用的主题（通常是最基础的或另一个主题）预设了pages类型。但Blowfish默认没有。所以当你创建content/pages/[myfirstlog.md](http://myfirstlog.md)时，Hugo虽然能生成文件，但Blowfish主题找不到对应的布局模板，就会用最基础的布局或者直接报错，导致页面显示异常

结论： 内容文件夹的名字（docs、posts、pages）不是随意的，它必须和主题的设计相匹配。对于Blowfish，常用的官方内容类型是docs（文档）和posts（博客文章）。

public文件夹是什么？为什么里面的结构和你的内容不一致？

  • public/文件夹是"成品仓库"：当你运行hugo命令（不带server）时，Hugo会把你的Markdown内容、主题模板、所有资源编译成最终的、纯静态的HTML/CSS/JS文件，并全部输出到这个文件夹。这就是你将来要上传到服务器（如GitHub Pages）的东西
  • 结构不一致的原因：Hugo在构建时，会根据配置文件和文章Front Matter（文件头部的YAML/TOML）对URL路径和文件结构进行重组。比如，一篇位于content/docs/getting-started/[index.md](http://index.md)的文章，最终在public/里可能会变成public/docs/getting-started/index.html。标题显示什么，也由Front Matter里的title字段决定，而不是文件名。所以，public/里的结构是编译后的结果，你不应该、也不需要直接去修改它

总结：问题与收获对照表
#

阶段你遇到的现象根本原因你学会的核心概念
1. 部署失败(SSH vs HTTPS)SSH连接超时，HTTPS成功网络代理配置是分协议的。你只为Git配置了HTTPS代理(http.proxy)，但SSH走的是另一套配置(~/.ssh/config)，且其代理工具/端口可能不正确Git的HTTPS协议在复杂网络下更可靠；SSH配置更复杂且依赖额外工具
2. CSS样式丢失线上页面无样式，但本地正常资源路径或完整性校验失败。可能是baseURL设置错误导致资源404，或SRI哈希值不匹配导致浏览器拒绝加载。你最终发现是Tailwind CSS框架未加载部署后需检查baseURL；浏览器的开发者工具(Network/Console)是排查前端问题的利器
3. 工作流分支困惑教程说用gh-pages，但你的工作流是pages分支名称是可配置的。关键是GitHub仓库设置中的"Pages"来源分支必须与工作流文件中deploy步骤推送的目标分支完全一致。叫gh-pages、pages或main都可以GitHub Actions工作流是脚本，你可以控制它把编译好的文件推送到哪个分支
4. .gitignore失效系统提示找不到文件Windows隐藏了文件扩展名。你创建的实际是.gitignore.txt，而Git只认无扩展名的.gitignore在Windows中创建以点开头的文件，最好在命令行中用echo > .gitignore命令
5. 主题子模块被修改themes/blowfish显示有未跟踪的修改你直接修改了主题内部的文件（如语言文件）。这违反了"覆盖原则"，导致子模块状态"不干净"永远不要修改themes/xxx/下的文件。自定义应通过复制文件到项目根目录的对应位置（如i18n/）来实现
6. 工作流构建错误defaultContentLanguage “en” does not match…Hugo配置文件语法或位置错误。可能是将语言配置错误地嵌套在了其他区块（如[markup]）内，或者格式有误（如给false加了引号）Hugo的配置文件(hugo.toml)对格式非常敏感。必须确保键值对位于正确的配置层级
7. 工作流部署错误需要Personal Access Token(PAT)GitHub安全策略：自2021年起，已禁止使用账户密码进行命令行认证，必须使用个人访问令牌或SSH密钥在GitHub设置中生成PAT，并在仓库的Secrets中设置为ACTIONS_DEPLOY_KEY或类似名称，供工作流使用

从经历中提炼的"生存法则"
#

  1. 环境配置是一切的基础：PATH、代理、权限，三者缺一不可
  2. 工具只是工具：blowfish-tools很方便，但理解Hugo的原理更重要
  3. 永远不要修改主题文件：用覆盖而不是修改
  4. Git子模块是动态链接：手动下载是静态副本，各有利弊
  5. 内容类型要匹配主题：docs vs pages，不是随便起名
  6. public是编译结果：不要直接修改，源文件在content
  7. SSH vs HTTPS：各有优劣，网络环境决定选择
  8. 开发者工具是神器：Network、Console能解决90%的前端问题
  9. 配置文件格式很严格：嵌套层级、引号、大小写都很重要
  10. GitHub Actions需要三者一致：工作流文件、分支名、Pages设置
